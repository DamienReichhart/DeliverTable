# Invoice Reductions — Design

**Date:** 2026-05-04
**Status:** Approved (spec)
**Owner:** damien.reichhart@gmail.com

## Problem

Customer invoices generated by the platform omit all order-level reductions. Three reduction sources already exist on `Order` (via the `OrderDiscount` collection): `Promotion`, `DiscountCode`, and `LoyaltyPoints`. None of them appear on the PDF, and the `InvoiceLine` model has no representation for them. As a result, the customer-facing invoice's `Total HT` / `Total TVA` / `Total TTC` reflect items at full price, while the actual amount the customer paid (`order.TotalAmount`) is lower. This is misleading to customers and incorrect for French VAT reporting.

There is also a latent bug: `OrderRepository.GetByIdWithFullDetailsAsync` — used by `InvoiceService.CreatePendingInvoicesAsync` — does not `Include(o => o.Discounts)`, so even if the invoice builder tried to read discounts, the collection would be empty.

## Goal

Customer invoices show every applied reduction as its own line, with HT/VAT/TTC values that produce a legally correct French invoice with correct totals.

## Non-goals

- Commission invoice changes (`BuildCommissionInvoice`). Commission is computed on `order.TotalAmount`, which is already post-discount, so reductions don't belong in the platform→restaurant invoice.
- Backfill of existing invoices. None have shipped to production with reductions visible (the bug above guaranteed `Discounts` was empty), so there's nothing to backfill.
- Any UI changes outside the PDF (no Blazor work).
- Loyalty points *earned*. Only points *used* are reductions; earnings are a separate accounting flow.

## Design decisions (recap)

| # | Decision | Rationale |
|---|---|---|
| Q1 | One line per `OrderDiscount`, listed below items, before totals | Most transparent for customers; legally clean (each reduction has a clear source); matches existing `OrderDiscount` data model; preserves original item prices for audit |
| Q2 | Proportional VAT allocation across mixed rates | Only option producing legally compliant French invoice with correct HT/VAT/TTC |
| Q3 | Customer invoice only | Commission already implicit in post-discount `TotalAmount` |

## Architecture

The system already has the right shape: `Invoice` has a collection of `InvoiceLine` rows, and totals are computed by summing line-level HT/VAT/TTC. We extend `InvoiceLine` with a `Kind` discriminator (`Item` | `Discount`). Discount lines store **negative** HT/VAT/TTC values, so the existing summation produces correct post-discount totals without any special casing.

```
OrderDiscount (Source, Amount-TTC)
        │
        ▼
BuildDiscountLines(items, discounts)   ← new pure helper in InvoiceService
        │
        ▼
InvoiceLine[] (Kind = Discount, negative values, one per (source × VAT rate))
        │
        ▼
invoice.Lines  ──────►  Sum(LineHt|LineVat|LineTtc)  ──►  TotalHt/Vat/Ttc
                              (unchanged)
        │
        ▼
InvoicePdfRenderer  ──►  Items table → Réductions table → Totals box
```

## Discount allocation algorithm

`BuildDiscountLines(IReadOnlyList<InvoiceLine> itemLines, IReadOnlyList<OrderDiscount> discounts) → IEnumerable<InvoiceLine>`

A pure private helper on `InvoiceService`. Called by `BuildCustomerInvoice` after item lines are constructed and before totals are summed.

### Per-discount steps

For each `OrderDiscount d`:

1. `subtotalByRate = itemLines.GroupBy(VatRate).ToDictionary(g => g.Key, g => g.Sum(l => l.LineTtc))`
2. `subtotalTtc = subtotalByRate.Values.Sum()`
3. If `subtotalTtc == 0`, skip this discount (defensive — orders always have items).
4. For each rate `r` with non-zero subtotal:
   - `share = subtotalByRate[r] / subtotalTtc`
   - `discountTtcForRate = round(d.Amount × share, 2, AwayFromZero)`
5. **Reconcile rounding drift**: the sum of per-rate slices may differ from `d.Amount` by a cent. Adjust the slice with the greatest absolute discount value (`|discountTtcForRate|`) by the drift so the slices sum exactly to `d.Amount`. Tie-break on rate descending for determinism.
6. For each rate `r` with non-zero slice, emit one `InvoiceLine`:
   - `Kind = InvoiceLineKind.Discount`
   - `LineTtc = -discountTtcForRate`
   - `LineHt  = round(LineTtc / (1 + r/100), 2, AwayFromZero)`
   - `LineVat = LineTtc - LineHt` *(no extra rounding — preserves `LineHt + LineVat == LineTtc`)*
   - `UnitPriceTtc = LineTtc`, `UnitPriceHt = LineHt`, `Quantity = 1`, `VatRate = r`
   - `Description`: `d.Description` (+ `" (TVA {r}%)"` suffix only if the discount produced more than one rate-line)
   - `SortOrder` continues from where item lines left off

### Edge cases

| Case | Handling |
|---|---|
| Discount > items subtotal | `OrderService.cs:100` already caps `discountAmount` at `originalAmount`. No special handling at invoice time. |
| `restaurant.IsVatRegistered = false` | All items have `VatRate = 0` → algorithm degenerates to a single-rate split with no VAT. No branching needed. |
| `subtotalTtc == 0` | Skip discount-line emission (defensive). |
| `order.Discounts` empty | Loop produces no discount lines — invoice identical to today. |
| Multi-rate split where one rate's slice rounds to 0 | Skip emitting that line; reconciliation step ensures totals still match. |

## Bug fix — load `Order.Discounts`

`DeliverTableInfrastructure/Repositories/OrderRepository.cs:99-107`

```csharp
public async Task<Order?> GetByIdWithFullDetailsAsync(int id, CancellationToken ct = default)
{
    return await _dbContext.Orders
        .Include(o => o.Customer)
        .Include(o => o.Restaurant)
        .Include(o => o.Items)
            .ThenInclude(i => i.Dish)     // ← new: required by BuildCustomerInvoice for item.Dish.VatRate
        .Include(o => o.Payments)
        .Include(o => o.Discounts)         // ← new: required by BuildDiscountLines
        .FirstOrDefaultAsync(o => o.Id == id, ct);
}
```

`ThenInclude(i => i.Dish)` is not present today even though `BuildCustomerInvoice` already reads `item.Dish.VatRate` — current invoice generation relies on whatever happens to be tracked in the `DbContext`. Making this explicit removes the latent fragility.

This fix lands in its own commit, before any of the new feature work, so the rest of the work is built on a correct foundation. A repository test asserts `result.Discounts.Count > 0` and `result.Items.All(i => i.Dish != null)` after seeding to prevent regression.

## PDF renderer

`DeliverTableWorker/Services/InvoicePdfRenderer.cs`

The renderer continues to iterate `invoice.Lines` in `SortOrder`, but groups visually by `Kind`:

- **Items section** (top): rows with `Kind = Item`, exactly as today.
- **Réductions section** (immediately below items, before totals): rows with `Kind = Discount`. Sub-header labeled `"Réductions"`; rows styled to indicate negative values (leading `−` and/or muted/red color, matching the renderer's existing visual vocabulary). Same column shape as the items table for alignment.
- **Totals box** (bottom): unchanged — already sums all lines, so HT/TVA/TTC reflect post-discount values automatically.

When `invoice.Lines.All(l => l.Kind == Item)` (no reductions), the Réductions section is suppressed entirely.

Localization: header `"Réductions"`. Column headers stay the same as the items table. Negative values render with a leading minus sign in the standard French currency format already used.
